## CV-1: S15 7-Step Compute Cycle — Sequence Validation

The 7-step sequence (Ingest → Trigger → Freeze → Validate Freshness → Compute → Atomic Write → Post-Commit Side Effects) is fundamentally sound and follows a well-established pattern analogous to database checkpoint cycles. The ordering dependencies are correctly captured: freeze must precede compute, freshness validation must gate compute (not follow it), and side effects must be post-commit so a crash during alerting never leaves the DB in a half-written state. One subtle dependency worth calling out: step 5 (Compute) has an internal ordering constraint — 4H bar synthesis must complete before EMA, EMA before Zone, Zone and Override before Policy Engine — and this should be enforced as a fixed pipeline, not parallel tasks, to avoid reading stale intermediate state within the same cycle.

Regarding SQLite WAL consistency during the freeze step: WAL mode does **not** automatically provide snapshot isolation across multiple separate SELECT statements. Each SELECT sees the database as of its start, but if the ingest process commits between two SELECTs in the freeze step, you get an inconsistent freeze. The fix is straightforward: wrap the entire freeze step in a single `BEGIN DEFERRED` transaction (read-only). In WAL mode, this establishes a consistent read snapshot at the first SELECT, and all subsequent reads within that transaction see the same database state. This is cheap — it does not block the ingest writer. Do **not** use `BEGIN IMMEDIATE` for the freeze, as that would unnecessarily acquire a write lock and block ingestion. The read transaction should be opened, all frozen values read into in-memory dataclasses, then the transaction closed before compute begins — so the freeze step holds the read snapshot only for the duration of the data copy, not the entire compute phase.

## CV-2: Evidence-Revision Model (analysis_version 1/2/3) — Implementation

The cleanest implementation is **three linked rows** in a dedicated `news_analysis` table, not three updates to the same `news_items` row. Each row shares a `news_item_id` foreign key and carries its own `analysis_version`, `timestamp`, `confidence`, and the full structured analysis JSON. This preserves the complete audit trail (you never lose what version 1 said when version 2 arrives), makes it trivial to query "what did the system know at time T" for replay testing, and avoids the semantic problem of mutating a row that a compute cycle already consumed. The `news_items` table holds the immutable raw message and Tier 1 scores; the `news_analysis` table holds the evolving intelligence. A convenience view or a `latest_analysis_version` column on `news_items` (updated on each new analysis insert) gives fast access to the current best assessment.

For the timing problem — Tier 2 analysis arriving after a compute cycle already used Tier 1 data — the answer is: **don't fight it, embrace it.** The compute cycle freezes whatever analysis version exists at freeze time. If only Tier 1 is available, the cycle uses Tier 1 scores (lower confidence, coarser classification). When the Tier 2 result lands 30-60 seconds later, it is written as `analysis_version=2` and picked up by the *next* 5-minute cycle. This is correct behavior, not a bug: the system's response to news is naturally staggered (Tier 1 immediate awareness → Tier 2 enriched understanding → AV/StockTwits cross-verification), and each compute cycle works with the best available information at its freeze point. The only case requiring special handling is `urgency: react_now` from Tier 2, which should trigger an emergency micro-cycle outside the normal 5-minute schedule — but this micro-cycle still follows the full 7-step sequence with its own `cycle_id`, it just fires on an event trigger rather than a timer.

## CV-3: Tier 2 Deterministic Mapper — Concrete Implementation

S15 is correct that the LLM must recommend, not act. The deterministic mapper should be a **rule-based classifier with explicit thresholds**, not a single confidence check. Concretely, it takes the Tier 2 structured JSON and produces typed candidate objects that the Policy Engine consumes like any other input. For `EventPenaltyCandidate`: if `event_penalty_trigger == true` AND `confidence >= 0.75` AND `category` is in the set `{macro_fed, macro_data, earnings, geopolitical}`, emit an `EventPenaltyCandidate(source="tier2_news", strength=confidence, category=category, news_item_id=id)`. The Policy Engine then decides whether to activate EventPenalty based on its own rules — for instance, it might require two independent sources confirming the event, or it might weight the candidate against the current Override state. Similarly, `GeoStressCandidate` fires only when `geostress_signal == true` AND `confidence >= 0.70` AND `category == "geopolitical"`. Below-threshold signals are still logged (for audit and replay) but do not produce candidates.

The key design principle is that **every threshold in the mapper is a named constant in the config_versions table**, not a hardcoded magic number. This lets you tune `TIER2_EVENT_PENALTY_CONFIDENCE_THRESHOLD` from 0.75 to 0.60 via config and immediately see the impact in veto analysis ("how many more EventPenalty activations would we have gotten last week at 0.60?"). The mapper itself should be a pure function: `(Tier2Result, MapperConfig) → List[Candidate]` with no side effects, no DB access, no network calls. This makes it trivially testable — feed it known Tier 2 outputs and assert on the candidates produced. The mapper is also where you handle Claude hallucination risk: if Tier 2 returns a category not in the allowed enum, or a confidence outside [0, 1], the mapper logs a schema violation, discards the result, and the system proceeds with Tier 1 data only. This is safer than letting a malformed LLM response propagate into framework state.
