## CV-4: ibapi (official) vs Alternatives â€” Practical Comparison

S15 is correct that `ib_insync` is archived (last meaningful commit mid-2024, no Python 3.12+ testing, author explicitly marked it unmaintained). The three realistic options are: (1) official `ibapi` from IB, (2) `ib_async` (a community fork/successor of ib_insync), and (3) a thin custom wrapper around `ibapi`. My recommendation is **option 3: official `ibapi` with a thin async wrapper you own**, and here is why.

The official `ibapi` is callback-based and synchronous â€” you subclass `EWrapper` for responses and call methods on `EClient` for requests. It is verbose but stable, ships directly from IB, and will never go unmaintained. The practical development cost is writing ~200-300 lines of wrapper code that: (a) runs the `ibapi` message loop in a dedicated thread, (b) bridges callbacks to `asyncio.Future` objects so your main engine can `await ib_bridge.get_vix()`, and (c) manages connection lifecycle (connect, disconnect, reconnect). This is a one-time cost that gives you full control and zero dependency risk. The `ib_async` fork (maintained by community contributors after ib_insync was archived) is tempting, but its maintenance bus factor is low and you would inherit the same risk that killed ib_insync. For your narrow use case â€” streaming 4 instruments, no order execution in Phase 1b â€” the wrapper is small enough that owning it is cheaper than debugging a third-party library's edge cases. A concrete skeleton: one `IBBridge` class with `connect()`, `disconnect()`, `subscribe_market_data(contract)`, `get_latest(symbol) -> dict`, and an internal `_run_loop()` thread. Contract definitions for VIX (IND/CBOE), VVIX (IND/CBOE), VX front-month (FUT/CFE), and CL front-month (FUT/NYMEX) are each ~5 lines. Total wrapper: under 300 lines, fully testable with a mock EWrapper.

## CV-5: IB Gateway Java Memory Tuning on VPS

S15's recommendation of 1024MB is a reasonable **starting point** but may be tighter than comfortable. IB Gateway's Java process has a base footprint of roughly 400-600MB for the GUI-less (headless) mode with IBC controller. Each streaming market data line adds negligible heap â€” your 4 instruments (VIX, VVIX, VX1!, CL) are trivial. The memory pressure comes from the JVM's garbage collector behavior: with `-Xmx1024m`, the GC will trigger more frequently as the heap fills, causing occasional latency spikes on the data callbacks. For a 4GB VPS running Python (~300-500MB for your engine + Telethon + PTB), SQLite (negligible), and IB Gateway, the realistic memory budget is: ~500MB OS/buffers + ~500MB Python processes + ~1024MB IB Gateway = ~2GB used, leaving ~2GB headroom.

My concrete recommendation: set `-Xmx768m -Xms512m` initially via the IBC configuration file (`javaheap=768` in `config.ini`). This is sufficient for your use case (4 streaming instruments, no order routing, no deep historical data requests) and gives Python more breathing room. Monitor with a simple cron job writing `ps -o rss` for the Java process to a log file, and bump to 1024m only if you observe OOM kills or excessive GC pauses. The IBC controller (`IBC.ini`) is where you set this â€” look for the `JavaHeapSize` or pass `-Xmx` via the `JAVA_OPTS` environment variable in your systemd unit's `Environment=` directive. One critical operational note: IB Gateway has a **mandatory daily restart** (default ~23:45 ET for US securities). On your VPS, configure IBC's auto-restart feature and ensure your systemd unit has `Restart=always` with `RestartSec=30` so the gateway comes back cleanly. Your `IBBridge` wrapper must handle the ~30-60 second reconnection window gracefully â€” mark IB data as STALE during this gap, and the compute cycle's freshness validation (step 4) will correctly degrade Override to STALE/UNKNOWN.

## CV-6: IB Pacing Rules and Streaming vs Polling

For your use case, **streaming is the correct approach, not polling**, and it largely sidesteps IB's pacing rules. IB's pacing violation rules primarily constrain *historical data requests* (no more than 60 requests in 10 minutes, no identical requests within 15 seconds) and *snapshot requests*. Streaming market data (`reqMktData`) is a different mechanism: you subscribe once and IB pushes updates to your callbacks. There is no per-tick rate limit on streaming â€” IB sends data as frequently as the exchange publishes it (typically every 250ms for liquid instruments). For your 4 instruments, you call `reqMktData` once per contract at startup and receive continuous updates indefinitely.

The practical setup: call `reqMktData(reqId=1, contract=vix_contract, genericTickList="", snapshot=False, ...)` for each of your 4 instruments on connection. Your `IBBridge` stores the latest values in an in-memory dict (e.g., `{"VIX": {"last": 18.5, "timestamp": ..., "bid": 18.4, "ask": 18.6}}`), updating on each `tickPrice`/`tickSize` callback. The compute cycle's freeze step simply reads these in-memory values â€” no network call, no pacing concern. The only pacing-relevant operation you might need is `reqHistoricalData` for backfilling bars after a gateway restart or reconnection, and here the 60-per-10-min rule applies. For 4 instruments requesting a few hours of 5-min bars, you are well within limits (4 requests, once per restart). One nuance: IB's "Lines of Market Data" limit depends on your account equity and subscription. A funded account typically gets 100 concurrent lines; you need only 4, so no issue. However, if you are using a paper trading account, verify the line limit â€” paper accounts sometimes have lower caps. Also note that VVIX may require the CBOE streaming data subscription to be active; without it, `reqMktData` for VVIX will return error code 354 ("Requested market data is not subscribed"). Confirm your IB market data subscriptions include: CBOE Streaming Bundle (for VIX + VVIX), CFE (for VX futures), and NYMEX Level 1 (for CL). This is the ~$10/mo cost S15 identified.
